# 工单全生命周期管理方案

> 从"生成一段文本"到完整的工单闭环

---

## 一、现状诊断

### 当前工单链路

```
用户问题 → 意图分类 → 条件分支(CASE1:投诉/退换货)
  → LLM-工单生成（输出 JSON 文本）
  → 工单确认 + 转人工引导（展示给用户）
  → 结束
```

### 缺失的能力

| 缺失项 | 说明 | 影响 |
|--------|------|------|
| 工单存储 | JSON 只展示给用户，没有持久化 | 无法追踪、无法统计 |
| 工单分配 | 不知道谁来处理 | 工单创建后无人跟进 |
| 状态流转 | 没有"待处理→处理中→已解决" | 用户不知道进度 |
| 处理通知 | 处理人不知道有新工单 | 响应延迟 |
| 时效监控 | 没有超时升级机制 | 工单可能被遗忘 |
| 关闭归档 | 没有"问题解决→知识回流" | 同样问题反复出现 |

## 二、工单全生命周期设计

### 2.1 生命周期图

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ 创建     │ → │ 分配     │ → │ 处理中   │ → │ 已解决   │ → │ 已关闭   │
│ (AI生成) │   │ (路由)   │   │ (人工)   │   │ (反馈)   │   │ (归档)   │
└─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘
                    │               │               │
                    ▼               ▼               ▼
              ┌─────────┐   ┌─────────┐   ┌─────────┐
              │ 超时升级  │   │ 用户追问  │   │ 知识回流  │
              │ (自动)   │   │ (重新打开) │   │ (→知识库) │
              └─────────┘   └─────────┘   └─────────┘
```

### 2.2 各阶段详细设计

#### 阶段 1：创建（AI 自动生成）

**触发条件：** 意图分类结果为"投诉"或"退换货"

**生成内容（已有，保持不变）：**
```json
{
  "summary": "用户申请退货退款，原因是尺码偏大",
  "category": "退换货",
  "priority": "中",
  "required_action": "核实订单→确认退货地址→发起退款",
  "order_id": "无"
}
```

**新增字段（产品化需要）：**

| 字段 | 说明 | 示例 |
|------|------|------|
| ticket_id | 唯一工单编号 | "TK-20260710-001" |
| created_at | 创建时间 | "2026-07-10T14:30:00" |
| user_id | 用户标识（脱敏） | "u_***1234" |
| source | 来源渠道 | "淘宝客服" |
| conversation_snapshot | 对话摘要（脱敏后） | "用户反映T恤尺码偏大..." |
| sla_deadline | 处理时效截止时间 | "2026-07-10T18:30:00" |

#### 阶段 2：分配（自动路由）

**路由规则：**

| 工单类型 | 分配对象 | 通知方式 |
|---------|---------|---------|
| 退换货（中优先级） | 售后客服组 | 飞书消息/企微通知 |
| 投诉（高优先级） | 客服组长 + 售后客服组 | 飞书消息 + @负责人 |
| 紧急（舆情/安全） | 客服主管 + 运营负责人 | 电话/短信 + 飞书 |

#### 阶段 3：处理中（人工跟进）

**客服操作：**
- 查看工单详情（对话摘要 + AI 分类）
- 更新处理状态（已联系用户 / 等待用户回复 / 等待上级审批）
- 添加处理备注
- 必要时转交其他人

#### 阶段 4：已解决（用户确认）

**关闭条件：**
- 客服标记"已解决" + 用户确认（或 24 小时无追问自动关闭）
- 用户满意度评价（可选）

#### 阶段 5：已关闭（归档 + 知识回流）

**归档后动作：**
- 统计入库（响应时长、处理时长、满意度）
- 高频问题自动标记 → 建议补充到知识库
- 典型案例 → 加入客服培训素材

## 三、MVP 落地方案：飞书多维表格

### 3.1 为什么选飞书多维表格？

| 对比维度 | 飞书多维表格 | 自建数据库 | Jira/其他工单系统 |
|---------|------------|-----------|------------------|
| 开发成本 | 零代码，30 分钟搭建 | 需要后端开发 | 需要对接 API |
| 通知能力 | 飞书原生通知 | 需要额外开发 | 需要对接 |
| 权限管理 | 飞书原生 | 需要自建 | 自带 |
| 数据看板 | 内置图表 | 需要前端开发 | 自带 |
| MVP 适用性 | ⭐⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |

### 3.2 表格结构设计

| 字段 | 类型 | 说明 |
|------|------|------|
| 工单编号 | 自动编号 | TK-{日期}-{序号} |
| 创建时间 | 日期 | 自动填入 |
| 工单类型 | 单选 | 退换货/投诉/物流异常/其他 |
| 优先级 | 单选 | 紧急/高/中/低 |
| 问题摘要 | 文本 | AI 自动生成 |
| 用户标识 | 文本 | 脱敏后 |
| 处理状态 | 单选 | 待分配/处理中/已解决/已关闭 |
| 处理人 | 人员 | 分配后填入 |
| 处理备注 | 文本 | 人工填写 |
| 响应时长 | 公式 | 首次响应时间 - 创建时间 |
| SLA 状态 | 公式 | 正常/即将超时/已超时 |

### 3.3 与 Dify 的对接方式

Dify 工作流 → HTTP 请求节点 → 飞书多维表格 API

```
Dify Chatflow:
  LLM-工单生成（生成 JSON）
    ↓
  Code 节点（构造飞书 API 请求体）
    ↓
  HTTP 请求节点（调用飞书多维表格 API 创建记录）
    ↓
  工单确认回复（展示给用户）
```

**飞书 API 创建记录示例：**

```json
POST https://open.feishu.cn/open-apis/bitable/v1/apps/{app_token}/tables/{table_id}/records
{
  "fields": {
    "工单类型": "退换货",
    "优先级": "中",
    "问题摘要": "用户申请退货退款，原因是尺码偏大",
    "处理状态": "待分配",
    "创建时间": "2026-07-10T14:30:00"
  }
}
```

## 四、工单数据看板

### 4.1 核心指标

| 指标 | 定义 | 目标值 |
|------|------|--------|
| 日均工单量 | 每天创建的工单数 | 监控趋势 |
| 平均响应时长 | 创建 → 首次人工回复 | < 30 分钟 |
| 平均处理时长 | 创建 → 已解决 | < 4 小时 |
| 超时率 | 超过 SLA 的工单占比 | < 5% |
| 满意度 | 用户评价 ≥ 4 分的占比 | ≥ 85% |
| 一次解决率 | 未重新打开的工单占比 | ≥ 80% |

### 4.2 飞书多维表格内置图表

- 工单量趋势图（按天）
- 工单类型分布（饼图）
- 处理人工作量分布（柱状图）
- SLA 超时率趋势

## 五、面试叙事模板

> "当前工单模块只生成了一段 JSON 文本展示给用户，是一个单向的输出。我设计了工单全生命周期管理方案，涵盖创建→分配→处理→解决→归档五个阶段。
>
> 对于 MVP 落地，我选择飞书多维表格作为轻量级工单系统——零代码搭建，30 分钟就能跑通。Dify 通过 HTTP 请求节点调用飞书 API 自动创建工单记录，处理人在飞书中收到通知并跟进，完成后数据自动归档。
>
> 更关键的是闭环设计：已解决的工单会按类型聚合，高频问题自动标记为'建议补充到知识库'——形成'用户问题→工单→知识'的飞轮，让系统越用越聪明。"
