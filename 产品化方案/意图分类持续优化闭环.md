# 意图分类持续优化闭环

> 从"手工补规则"到 Bad Case 驱动的自动化迭代体系

---

## 一、现状诊断

### 1.1 当前迭代方式

```
发现 Bad Case → 人工分析 → 手写新规则 → 加到 Prompt
```

### 1.2 这种方式的致命缺陷

| 问题 | 说明 | 后果 |
|------|------|------|
| 规则膨胀 | 每次迭代加 2-3 条规则，V10 可能有 30+ 条 | 规则冲突、维护成本指数增长 |
| 无冲突检测 | A 规则和 B 规则可能矛盾 | 模型困惑，准确率反而下降 |
| 无回归测试 | 修了 A 问题可能破坏 B 问题 | "打地鼠"式迭代 |
| 无优先级排序 | 不知道先修哪个 Bad Case | 精力花在低频问题上 |
| 无效果归因 | 改了 Prompt 后准确率变了，但不知道为什么 | 无法积累方法论 |

### 1.3 当前 Prompt 规则列表（潜在风险）

```
- "询问材质/颜色/尺码/功能/价格" → 售前咨询
- 只有明确表达愤怒/不满/要求赔偿 → 投诉
- "xxx好吗/怎么样/好不好" → 售前咨询
- 纯疑问句不带情绪 → 优先售前咨询
- "退货/换货/退款"关键词 → 退换货
- "物流/发货/快递到哪/订单状态" → 订单查询
```

> ⚠️ 这些规则之间已经存在潜在冲突。比如用户说"这衣服质量太差我想退款"，同时命中"质量→投诉"和"退款→退换货"两条规则。

## 二、Bad Case 驱动的优化闭环

### 2.1 闭环全景图

```
  ┌─────────────────────────────────────────────────────┐
  │                                                       │
  ▼                                                       │
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 线上运行  │ → │ Bad Case │ → │ 根因分析  │ → │ Prompt   │
│          │   │ 自动采集  │   │ + 分类    │   │ 迭代     │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
     ↑                                              │
     │              ┌──────────┐                    │
     └──────────────│ 回归测试  │←───────────────────┘
                    └──────────┘
```

### 2.2 各环节设计

#### 环节 1：Bad Case 自动采集

**采集来源：**

| 来源 | 触发条件 | 采集内容 |
|------|---------|---------|
| 转人工 | 用户主动转人工 | 转人工前最后 3 轮对话 + AI 分类结果 |
| 用户否定 | 用户回复"不对""不是""你说错了" | 对话上下文 + AI 分类结果 |
| 低置信度 | confidence < 0.6 | 原始 Query + 分类结果 |
| 用户投诉 | 用户表达不满 | 对话上下文 |

**采集数据格式：**

```json
{
  "bad_case_id": "BC-20260710-001",
  "timestamp": "2026-07-10T14:30:00",
  "source": "转人工",
  "user_query": "这件衣服质量太差了我要退款",
  "ai_intent": "投诉",
  "ai_confidence": 0.88,
  "ai_reason": "包含'质量太差'表达不满",
  "expected_intent": "退换货",
  "correct_reason": "用户核心诉求是退款而非投诉，应走退换货流程",
  "context": ["前一轮对话..."],
  "status": "待分析"
}
```

#### 环节 2：根因分析 + 分类

**Bad Case 分类体系：**

| 类型 | 描述 | 示例 | 优先级 |
|------|------|------|--------|
| **边界混淆** | 两个意图之间边界模糊 | "质量差想退款"→ 投诉 vs 退换货 | 🔴 高 |
| **规则缺失** | 新出现的表达方式未被覆盖 | "这个踩雷了"（=不满意） | 🔴 高 |
| **规则冲突** | 两条规则给出相反结论 | 同时命中投诉和退换货规则 | 🟡 中 |
| **关键词误判** | 关键词匹配过于粗暴 | "这衣服投诉率低吗"（售前咨询，非投诉） | 🟡 中 |
| **情绪误判** | 把中性表达当负面情绪 | "怎么样"被误判为不满 | 🟢 低 |
| **模型幻觉** | 模型输出与 Prompt 规则矛盾 | Prompt 明确写了规则，模型没遵守 | 🟢 低 |

**优先级判定矩阵：**

| | 高频（>5次/周） | 中频（2-5次/周） | 低频（<2次/周） |
|---|---|---|---|
| **高影响**（涉及资损/舆情） | 🔴 P0 立即修复 | 🔴 P0 | 🟡 P1 |
| **中影响**（用户体验下降） | 🟡 P1 | 🟡 P1 | 🟢 P2 |
| **低影响**（轻微不准确） | 🟢 P2 | 🟢 P2 | ⚪ P3 暂缓 |

#### 环节 3：Prompt 迭代

**迭代原则：**

1. **先减后加**：修改前先评估是否能删除已有规则，避免规则膨胀
2. **加示例代替加规则**：在 Prompt 中加入 2-3 个典型 Bad Case 示例，比加抽象规则更有效
3. **一次只改一个点**：每次迭代只修复 1 类问题，便于归因
4. **必有回归测试**：每次改 Prompt 必须跑全量 100 条测试用例

**迭代记录模板：**

```markdown
## 迭代 #3 - 2026-07-10

**修复问题：** "质量差想退款" 被误判为投诉
**Bad Case 数量：** 本周出现 8 次
**根因：** "质量差"触发了投诉规则，但用户实际诉求是退款（退换货）
**修改内容：**
- 新增规则：当同时出现"质量描述+退款诉求"时，优先判为退换货
- 新增示例：用户说"质量太差想退款" → 退换货
**回归测试结果：** 100 条用例，准确率 90% → 92%
**副作用检查：** 真正投诉场景（"质量太差我要投诉你们"）仍正确分类
```

#### 环节 4：回归测试

**每次 Prompt 变更必须跑：**

| 测试集 | 用例数 | 通过标准 |
|--------|--------|---------|
| 全量回归集 | 100 条 | 准确率 ≥ 变更前 |
| Bad Case 专项集 | 本次修复相关的 5-10 条 | 100% 通过 |
| 易误判集 | 边界场景 10 条 | 准确率 ≥ 90% |

## 三、意图分类体系的长期优化方向

### 3.1 6 类意图是否合理？

**当前潜在问题：**

| 意图对 | 边界模糊场景 | 建议 |
|--------|------------|------|
| 退换货 vs 投诉 | "质量太差我要退款" | 优先级规则：有退款诉求→退换货；纯情绪表达→投诉 |
| 订单查询 vs 退换货 | "快递太慢了我要退" | 识别核心动作：退→退换货；查→订单查询 |
| 售前咨询 vs 其他 | "你们公司多少人" | 明确"商品相关"vs"非商品相关"边界 |

### 3.2 未来可考虑的优化方向

| 方向 | 说明 | 适用阶段 |
|------|------|---------|
| Few-shot 示例 | 在 Prompt 中加入每类意图的典型示例 | 短期（立即可做） |
| 子意图细分 | 售前咨询 → 材质/尺码/价格/库存 子类 | 中期（数据积累后） |
| 微调小模型 | 用标注数据微调分类模型，替代 Prompt | 长期（标注量 > 1000 条后） |

## 四、面试叙事模板

> "我的意图分类 V1.0 只有基础 Prompt，发现'这件T恤是什么材质'被误判为投诉。V2.0 加了判断规则解决了，但我意识到手工补规则的方式到 V10 就会变成一坨乱麻——规则冲突、维护成本指数增长。
>
> 所以我设计了一套 Bad Case 驱动的优化闭环：线上自动采集转人工/用户否定/低置信度等 Bad Case → 按边界混淆/规则冲突/关键词误判等维度分类 → 按频率×影响度排优先级 → 每次只修 1 类问题 + 全量回归测试。
>
> 这样就把意图分类的迭代从'拍脑袋加规则'变成了有数据、有优先级、可度量的工程化流程。"
