# 知识库可持续运营方案

> 从"手动 Markdown + 关键词铺底"到可规模化运营的知识库体系

---

## 一、现状诊断

### 1.1 当前方案回顾

| 方案 | 做法 | 解决的问题 | 引入的新问题 |
|------|------|-----------|-------------|
| V1.0 | 商品详情 Markdown 单文件 | 基础问答 | 语义检索长尾词命中率低 |
| V1.5 | 关键词铺底策略 | 搜索"材质"有结果了 | 人工维护不可持续 |

### 1.2 核心矛盾

> 当商品 SKU 从 4 个扩展到 40 个、100 个时，手动维护关键词铺底的工作量指数级增长，而召回率提升边际递减。

### 1.3 产品经理必须回答的三个问题

1. **新品上线怎么同步到知识库？** 谁来操作？多久一次？
2. **怎么知道知识库有没有覆盖用户的问题？** 覆盖率是多少？
3. **怎么发现知识过期或错误？** 用户反馈怎么回流？

## 二、知识库三层架构

```
                    ┌──────────────────────────┐
                    │   第三层：动态知识        │
                    │   自动生成 / 实时更新     │
                    │   商品属性同步 + 高频QA   │
                    ├──────────────────────────┤
                    │   第二层：结构知识        │
                    │   模板化 / 半自动维护     │
                    │   商品FAQ + 类目通用     │
                    ├──────────────────────────┤
                    │   第一层：基础知识        │
                    │   手动维护 / 稳定不变     │
                    │   售后政策 + 品牌介绍     │
                    └──────────────────────────┘
```

| 层级 | 内容 | 更新频率 | 维护方式 | 占比 |
|------|------|---------|---------|------|
| 第一层：基础知识 | 售后政策、退换货规则、物流说明、品牌介绍 | 季度/政策变更时 | 运营手动维护 | ~20% |
| 第二层：结构知识 | 商品FAQ（材质/尺码/颜色问答对）、类目通用话术 | 新品上线时 | 模板自动生成+人工审核 | ~50% |
| 第三层：动态知识 | 商品属性实时同步、未命中Query聚类生成、促销活动信息 | 实时/每日 | 系统自动 | ~30% |

## 三、各层实施方案

### 3.1 第一层：基础知识（当前已有，优化维护流程）

**已有内容：** 售后政策知识库、通用FAQ

**优化方向：**

| 当前 | 优化后 |
|------|--------|
| 无变更记录 | 每次修改记录时间+修改人+变更内容 |
| 无过期提醒 | 设置季度 review 提醒（如退换货政策变更） |
| 无版本管理 | Git 管理 Markdown，可回滚 |

### 3.2 第二层：结构知识（新增：商品 FAQ 模板化生成）

**核心理念：** 用模板自动生成商品 FAQ，而不是手动为每个商品写文档。

**模板示例（以服装类目为例）：**

```markdown
# {商品名称}-商品FAQ

## 材质相关
- Q: {商品名称}是什么材质？
  A: {材质成分}，{材质特点描述}
- Q: {商品名称}会起球吗？
  A: {起球说明}
- Q: {商品名称}会缩水吗？
  A: {缩水说明}

## 尺码相关
- Q: {商品名称}偏大还是偏小？
  A: {版型说明}，建议{尺码建议}
- Q: 身高{xxx}体重{xxx}穿什么码？
  A: 参考尺码表，建议{推荐尺码}

## 颜色相关
- Q: {商品名称}白色会透吗？
  A: {透度说明}
- Q: {商品名称}实物和图片有色差吗？
  A: {色差说明}
```

**模板变量来源：**

| 变量 | 来源 | 示例 |
|------|------|------|
| 商品名称 | 商品数据库 | "纯棉宽松T恤" |
| 材质成分 | 商品属性 | "100%新疆长绒棉" |
| 版型说明 | 商品属性 | "宽松版型，落肩设计" |
| 起球说明 | 类目默认值+特殊说明 | "精梳工艺处理，不易起球" |

**效果：**
- 一个类目 1 个模板 → 覆盖该类目下所有商品
- 运营只需填写变量值，不需要写整段 FAQ
- 100 个 SKU 的知识库维护成本 ≈ 维护 1 个模板 + 100 行商品属性表

### 3.3 第三层：动态知识（远期规划，设计思路）

**3.3.1 商品属性自动同步**

```
商品数据库（ERP/电商后台）
  ↓ 定时同步（每日/每次上新）
知识库更新
  ↓
商品属性 → 自动生成 FAQ 问答对
```

**3.3.2 未命中 Query 自动聚类 → 生成候选知识**

这是解决"知识库覆盖率"问题的核心闭环：

```
用户问题 → 知识检索 → 未命中（score < 0.4）
  ↓
收集到"未命中池"
  ↓
每周自动聚类（相似问题归为一组）
  ↓
运营审核 → 确认为有效知识需求 → 生成新知识条目
  ↓
加入知识库 → 下次可命中
```

**示例：**

| 未命中 Query | 聚类结果 | 新增知识 |
|-------------|---------|---------|
| "能穿着睡觉吗" | 穿着场景类 | "T恤面料柔软亲肤，可作为睡衣穿着" |
| "胖子能穿吗" | 大码适配类 | "最大码XXX，适合体重XXX以内" |
| "送男朋友合适吗" | 送礼场景类 | "简约百搭款，适合送礼，支持礼品包装" |

## 四、知识库质量度量

### 4.1 覆盖率指标

| 指标 | 定义 | 目标值 | 当前值 |
|------|------|--------|--------|
| 语义检索命中率 | 检索返回非空结果的请求占比 | ≥ 80% | ~80%（关键词铺底后） |
| 有效回答率 | 用户未转人工且未追问的对话占比 | ≥ 70% | 未测量 |
| 知识条目覆盖率 | 已覆盖的意图子类别 / 全部意图子类别 | ≥ 90% | ~75% |
| 知识新鲜度 | 最近 30 天内有更新的知识条目占比 | ≥ 20% | ~0%（无更新机制） |

### 4.2 质量巡检清单（每周）

| 检查项 | 频率 | 负责人 |
|--------|------|--------|
| 未命中 Top 10 Query 聚类分析 | 每周 | 运营 |
| 已命中但回答不准确的 Bad Case 复查 | 每周 | 产品 |
| 退换货/投诉场景知识是否最新 | 每月 | 运营 |
| 商品信息与知识库一致性抽查 | 每月 | 运营 |

## 五、从 MVP 到规模化的路径

| 阶段 | 商品数 | 维护方式 | 人力投入 |
|------|--------|---------|---------|
| **MVP**（当前） | 4 个商品 | 手动写 Markdown | 1 小时/商品 |
| **V2.0** | 20-50 个商品 | 模板自动生成 + 人工审核 | 10 分钟/商品 |
| **V3.0** | 100+ 个商品 | 商品数据库自动同步 + 未命中闭环 | 5 分钟/周（审核聚类结果） |

## 六、面试叙事模板

> "我在 V1.0 阶段发现知识库召回存在长尾词命中率为 0 的问题，比如搜索'材质'无法命中语义检索。我设计了关键词铺底作为紧急方案，但同时也认识到这不可持续——当 SKU 从 4 个扩展到 100 个时，人工维护会崩溃。
>
> 所以我规划了三层知识架构：第一层基础政策手动维护（占比 20%），第二层商品 FAQ 用模板自动生成（占比 50%），第三层通过未命中 Query 自动聚类生成候选知识（占比 30%）。核心思路是让知识库从'纯手工'变成'系统自动生成 + 人工审核'，把人力从写文档变成做审核。
>
> 同时定义了四个覆盖率指标来度量知识库质量，让迭代有数据依据而不是拍脑袋。"
