## 一、核心指标体系

### 北极星指标
**自助解决率** = AI独立解决的会话数 / 总会话数

### 一级指标

| 指标 | 计算方式 | 目标值 | 监控频率 |
|------|----------|--------|----------|
| 自助解决率 | AI解决会话/总会话 | ≥ 60% | 每日 |
| 意图识别准确率 | 正确分类/总分类 | ≥ 90% | 每周 |
| 知识库召回率 | 有结果检索/总检索 | ≥ 80% | 每周 |
| 转人工率 | 转人工会话/总会话 | ≤ 40% | 每日 |
| 用户满意度 | 满意评价/总评价 | ≥ 4.0/5.0 | 每周 |

### 二级指标

| 指标 | 说明 | 作用 |
|------|------|------|
| 意图识别准确率 | 正确分类/总分类 | 评估意图分类 Prompt |
| 知识库命中率 | result不为空/总检索 | 评估知识库覆盖度 |
| 平均响应时间 | 用户提问到回答的时间 | 评估性能体验 |
| Bad Case率 | 转人工+解析失败/总会话 | 评估问题会话占比 |
| JSON 解析成功率 | 成功解析/总分类 | 评估 Code 节点容错 |
| 误判率 | 正常问题被分到高危/投诉 | 评估分类边界 |

## 二、实战踩坑 & 解决方案汇总

> 这是本项目最有价值的部分——从0到1搭建过程中的真实问题与解决方案

### 坑1：意图分类误判
| 项目 | 内容 |
|------|------|
| 现象 | `T恤是什么材质` 被分类为"投诉"，置信度0.92 |
| 根因 | Prompt 未区分"询问信息"和"表达不满"，DeepSeek 看到"材质"联想到质量问题 |
| 方案 | Prompt 增加"关键判断规则"：明确"询问材质/颜色/尺码→售前咨询" |
| 效果 | 误判率从80%降至5% |

### 坑2：DeepSeek <think> 标签污染
| 项目 | 内容 |
|------|------|
| 现象 | JSON 解析失败，confidence 恒为0 |
| 根因 | DeepSeek V3 输出带 `<think>推理过程</think>` 标签，Code节点无法解析 |
| 方案 | Code节点增加正则 `re.sub(r'<think>.*?</think>', '', text)` 清理 |
| 效果 | 解析成功率 0% → 95% |

### 坑3：Max Token 不足导致 JSON 截断
| 项目 | 内容 |
|------|------|
| 现象 | finish_reason=length，text 只有 <think> 没有 JSON |
| 根因 | MaxToken=256，<think> 推理过程耗尽了所有 Token |
| 方案 | MaxToken 256 → 512 |
| 效果 | 截断问题解决 |

### 坑4：知识检索 Score 阈值过高
| 项目 | 内容 |
|------|------|
| 现象 | 知识检索 result 恒为空数组 [] |
| 根因 | Score阈值=0.6，但实际召回分数0.53~0.56，全部被过滤 |
| 方案 | Score阈值 0.6 → 0.4 |
| 效果 | 召回率 0% → 正常 |

### 坑5：Dify 条件分支语法差异
| 项目 | 内容 |
|------|------|
| 现象 | 配置"等于"选项不存在 |
| 根因 | Dify 条件分支用"是"而不是"等于" |
| 方案 | 条件选"是"代替"等于" |

### 坑6：变量引用语法错误
| 项目 | 内容 |
|------|------|
| 现象 | 提示"无效的变量" |
| 根因 | 手写变量名而非用变量选择器，格式不匹配 |
| 方案 | 用 Dify 变量选择器点选，不用手写 |

## 五、埋点方案（落地版）

### 5.1 落地方式：Dify 工作流内 Code 节点 + Webhook

由于 Dify Chatflow 没有内置埋点系统，在每个关键节点后增加一个 **Code 节点**，将关键数据通过 HTTP 请求发送到飞书多维表格或自建日志服务。

**埋点架构：**

```
[意图分类 LLM] → [埋点 Code] → Webhook → 飞书多维表格
[知识检索]     → [埋点 Code] → Webhook → 飞书多维表格
[生成回答]     → [埋点 Code] → Webhook → 飞书多维表格
[转人工]       → [埋点 Code] → Webhook → 飞书多维表格
```

### 5.2 关键事件埋点

| 事件名 | 触发时机 | 上报参数 |
|--------|----------|----------|
| session_start | 用户开始对话 | user_id, timestamp, source |
| intent_classify | 意图分类完成 | intent, confidence, query |
| code_parse | JSON解析 | success, error_reason |
| knowledge_search | 知识库检索 | query, result_count, top_score |
| answer_generate | 生成回答 | intent, has_knowledge |
| transfer_human | 转人工 | reason（低置信/高危意图/检索未命中）|
| ticket_create | 生成工单 | category, priority |
| session_end | 会话结束 | duration, turn_count, resolution |

### 5.3 埋点 Code 节点示例

```python
import json
import requests

def main(intent: str, confidence: float, query: str) -> dict:
    payload = {
        "event": "intent_classify",
        "intent": intent,
        "confidence": confidence,
        "query": query,
        "timestamp": ""  # Dify 会自动注入
    }
    
    # 发送到飞书多维表格 Webhook
    # webhook_url = "https://open.feishu.cn/..."
    # requests.post(webhook_url, json=payload)
    
    return {"logged": True}
```

## 七、线上监控与告警体系

### 7.1 离线评测 vs 线上监控

| 维度 | 离线评测（已有） | 线上监控（新增） |
|------|----------------|-----------------|
| 数据来源 | 100 条手工测试用例 | 真实用户对话 |
| 数据分布 | 均匀分布（每类意图 15-20 条） | 自然分布（售前 65%，其他 35%） |
| 发现问题 | 已知问题（主动设计） | 未知问题（长尾、新表达） |
| 时效性 | 版本发布时 | 实时 |
| 核心价值 | 验证功能正确性 | 发现线上异常 |

### 7.2 线上监控大盘（飞书多维表格看板）

**实时监控指标：**

| 指标 | 计算方式 | 告警阈值 | 告警动作 |
|------|---------|---------|---------|
| 转人工率飙升 | 最近 1 小时 vs 过去 24 小时均值 | > 2 倍 | 飞书通知产品+运营 |
| JSON 解析失败率 | 解析失败次数 / 总分类次数 | > 10% | 飞书通知产品 |
| 知识检索空结果率 | result 为空 / 总检索次数 | > 30% | 飞书通知运营 |
| 单次对话轮次异常 | 单会话 > 10 轮 | > 5% 会话 | 标记待复查 |
| 意图分布偏移 | 某类意图占比波动 > 50% | 持续 1 小时 | 飞书通知产品 |

**每日数据看板（T+1 自动生成）：**

| 模块 | 指标 | 可视化 |
|------|------|--------|
| 流量概览 | 总会话数、总消息数、人均轮次 | 数字卡片 |
| 意图分布 | 6 类意图占比 | 饼图 |
| 处理结果 | 自助解决 / 转人工 / 工单生成 占比 | 堆叠柱状图 |
| 质量趋势 | 意图准确率、检索命中率 7 日趋势 | 折线图 |
| Bad Case Top 10 | 转人工最多的 Query | 排行榜 |
| 成本 | Token 消耗趋势 | 折线图 |

### 7.3 异常告警流程

```
监控指标触发阈值
  ↓
飞书机器人自动发送告警卡片
  ↓
  ├─ P0（转人工率>2倍/解析失败>10%）→ 产品 5 分钟内响应
  ├─ P1（检索空结果>30%/轮次异常）  → 运营 30 分钟内排查
  └─ P2（意图分布偏移）              → 次日周报中体现
```

### 7.4 与离线评测的联动

```
线上监控发现异常
  ↓
抽样分析 Bad Case
  ↓
转化为离线测试用例（加入回归测试集）
  ↓
Prompt 迭代修复
  ↓
离线回归测试通过
  ↓
上线 → 线上监控验证
```

这样形成了**"线上发现 → 离线复现 → 修复 → 回归 → 上线验证"**的完整闭环。

## 八、项目亮点总结（面试用）

| 亮点 | 说明 |
|------|------|
| 多级路由 | 意图分类→风险分级→自动处理/工单/转人工，不是简单问答匹配 |
| 工单闭环 | 高危问题自动生成结构化工单，可对接飞书/Jira等工单系统 |
| Prompt 工程 | 记录 V1.0→V2.0 完整迭代，体现产品思维 |
| 容错设计 | Code节点4层容错，处理 `<think>`/代码块/格式异常 |
| 数据驱动 | 100条测试用例 + 量化指标体系 + Bad Case 分析 |
| 可落地 | Dify 工作流可视化编排，运营可维护，支持私有部署 |
| 线上可观测 | 埋点→飞书看板→异常告警→Bad Case 回流测试集 全链路闭环 |
